# 通用工作流引擎（原子記憶 V2.8）

> 全域自動載入指令。專案特有知識由各專案根目錄 `CLAUDE.md` 定義。

## 一、_AIDocs 知識庫

### Session 啟動檢查

**每個 session 第一次互動前**，檢查專案根目錄是否有 `_AIDocs/`：

- **沒有** → 執行 `/init-project` skill
- **已有** → 啟用工作中維護規則

### 工作中維護規則

1. **開工前必讀**：先讀 `_AIDocs/_INDEX.md` 確認是否已有相關文件
2. **禁止憑記憶作業**：不確定的架構事實必須查文件或原始碼
3. **活文件更新**：修改核心結構、發現新認知、踩到陷阱 → 更新 `_AIDocs/*.md` + `_CHANGELOG.md`
4. **新增文件時**：同步更新 `_AIDocs/_INDEX.md`

## 二、原子記憶系統

原子記憶分**全域層**（`~/.claude/memory/`）和**專案層**（`projects/{slug}/memory/`），每層各有 `MEMORY.md`（Atom Index + 高頻事實）。

Hook 自動處理：embedding、向量搜尋、知識萃取、跨 session 鞏固、注入。Claude 負責的是**決策**——決定記什麼、分什麼類、怎麼演進。

> 完整規格：`~/.claude/memory/SPEC_Atomic_Memory_System.md`

### 決策記憶三層分類

| 符號 | 說明 | 引用行為 |
|------|------|---------|
| `[固]` | 跨多次對談確認，長期有效 | 直接引用 |
| `[觀]` | 已決策但可能演化 | 觸及時簡短確認 |
| `[臨]` | 單次決策 | 觸及時明確確認 |

### 主動寫入時機

1. **使用者明確要求**：「記住」「以後都這樣」→ 直接 [固]
2. **使用者做了取捨**：選 A 不選 B → [臨]，附推測原因
3. **發現反覆模式**：跨 2+ 次互動的相同偏好 → [觀]
4. **踩到陷阱**：debug 超過 10 分鐘 → 寫入 failures atom
5. **架構決策**：影響多檔案的設計選擇 → 寫入 decisions atom
6. **工具/環境知識**：確認的版本、路徑、設定值 → 寫入對應 atom

不寫入：臨時嘗試方案、未經確認的猜測、單次 session 不太可能復現的細節。

### 分類演進（LLM 責任）

- **觸及 [臨] 決策**：簡短提及 + 確認 → Confirmations +1
- **Confirmations ≥ 2 的 [臨]**：建議晉升 → [觀]
- **Confirmations ≥ 4 的 [觀]**：建議晉升 → [固]（需使用者確認）
- **使用者推翻已記錄決策**：更新 atom，降級或標記 Supersedes
- **矛盾偵測**：新決策與已有 atom 矛盾時，標出衝突，問使用者取捨
- 淘汰：超期 atom 移入 `_distant/{年}_{月}/`（遙遠記憶），不刪除

### 注入與引用原則

- 精確度 > token 節省。寧可多載入，不猜測跳過
- 已載入但不相關的 atom：靜默忽略
- 引用已記錄事實時直接引用，不重新分析原始碼

### 管理原則

- `MEMORY.md` ≤30 行（索引+高頻事實），atom 檔 ≤200 行
- `_CHANGELOG.md` 保留 ~8 筆，舊條目移至 `_CHANGELOG_ARCHIVE.md`
- 健檢：`python ~/.claude/tools/memory-audit.py`

## 三、工作結束同步

完成有意義的修改後，主動向使用者提出同步：

> 「這次修改涉及 N 個檔案，要我同步更新 {適用項目} 嗎？」

| 條件 | 同步步驟 |
|------|---------|
| 有 `_AIDocs/` | → 追加 `_CHANGELOG.md`（超 8 筆觸發滾動淘汰） |
| 有新知識/決策/坑點 | → 更新 atom 檔（知識段落 + Last-used） |
| 有 `.git/` | → 秘密洩漏檢查 → `git add` → `git commit` → `git push` |
| 有 `.svn/` | → 秘密洩漏檢查 → `svn add` → `svn commit` |
| 都沒有 | → 僅更新 memory atoms |

適用的步驟都要做完，不要只做一半。

**Workflow Guardian**（`workflow-guardian.py`）自動追蹤修改，未同步時會阻止結束。同步完成後發 `workflow_signal: sync_completed` 解除閘門。

## 五、對話管理

### 拆分指引

- 獨立子任務可新開對話（MEMORY.md 自動載入）
- 拆分前確保：新發現已寫入 _AIDocs、重要事實已存入 atom
- 有順序依賴的任務應在同一對話完成

### 主動續航

1. **段落完成即存**：完成一段工作後立即將進度寫入 atom
2. **Token 上限預警**：快碰上限時優先存檔工作狀態（任務、進度、下一步、阻塞點）
3. **重試追蹤**：反覆修正場景 → 記錄重試次數+調整+成敗原因，避免跨 session 重走錯路
4. **自動續接**：`/resume` skill — 生成續接 prompt → MCP 自動化開新 VS Code 視窗 → 貼上 → 執行

### 工作單元命名

有價值的修改、指示、洞察，賦予簡短中文名稱追蹤狀態（🔄→✅/❌/⏸），可跨 session 引用。

### 自我迭代（V2.6）

記憶系統自身隨使用演進，追求精確度/協助力/良善性三維平衡。8 條核心規則：

1. **品質函數**：確認(+) → Confirmations+1；糾正(−) → 更新規則；無回饋 → 不動作
2. **收斂優先**：新增前先檢查可合併的既有規則，規則總數趨向收斂
3. **證據門檻**：≥2 次獨立 session 觀察才建立正式規則，單次只記 [臨]
4. **淘汰勇氣**：每新增 1 條，檢查是否有 1 條可淘汰（Via negativa）
5. **震盪偵測**：同一 atom 3 session 內改 2+ 次 → 暫停，等更多證據
6. **成熟度感知**：穩定期/成熟期減少新規則，專注精煉（hook 自動報告）
7. **三維平衡**：精確度/協助力/良善性不偏廢
8. **演進邊界**：只改行動/知識段落，**絕對禁止**自改 Confidence 層級

> 理論背書詳見 `memory/openclaw-self-iteration.md`

**定期檢閱**：收到 SessionStart 提醒時，掃描近期 episodic atoms，收攏重複模式為 [觀]/[固]，完成後寫入 `workflow/last_review_marker.json` 重置計數器。

### 提醒開新 Session

Context 被壓縮、任務告一段落、回應明顯變慢 → 提醒使用者開新 session（確保新知識已存入 atom）。

### Token 節省

- 已決策事項簡短提及，不重複分析
- 已有 _AIDocs 文件的內容直接引用，不重新掃描原始碼
- 重複性修改先改 1-2 個確認模式，再批量執行
