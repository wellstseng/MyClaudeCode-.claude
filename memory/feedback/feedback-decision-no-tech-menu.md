## 知識

- [臨] 需要使用者決策時不要列技術方案選單讓他挑（工具名、協定、機制都不是使用者該選的層級）。
- [臨] 我心裡可以做比較表，輸出前收起來；預設只講「我建議 X，因為 Y（白話一句）」然後直接執行。
- [臨] Why：使用者明確回饋「你丟出來一堆我看不懂的技術知識還成為選項，我就會想要乾脆直接推翻」。把技術細節攤給非該領域專家＝強迫他們選沒判斷依據的東西，反而讓他們想拒絕整個建議。這違反 USER.md「詳細綜觀 + 分析條列 + 建議優選」的重點在「優選」不是「攤開」。
- [臨] 例外：使用者問「為什麼」或有真實業務 tradeoff（成本/時間/風險，不是技術名詞）才展開細節。技術選擇（WebFetch vs Playwright vs curl、DB、framework）由我選。
- [臨] 反例：對要登入網頁這件事，列了 WebFetch/Playwright/curl 三行比較表讓使用者在「能不能登入」「能不能到內網」「開不開視窗」間挑 — 使用者不需要也不想知道。
- [臨] 正例：「我換個做法，不用開瀏覽器視窗，從你電腦背景直接去問網頁就行。Phase 2 就這樣做。」

## 行動

- 需要決策 → 先問自己：這是使用者該選的（業務 tradeoff）還是我該選的（技術實作）
- 若是技術實作 → 直接推薦一個並說白話理由，不列選單
- 若是業務 tradeoff → 一句建議 + 一句白話優選理由 + 條列關鍵差異（最多 3 行，不帶技術名詞）
- 自己列的比較表留在思考階段，不輸出給使用者
